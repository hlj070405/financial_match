## 第4页 · 技术现状与挑战

> 展示现有技术方案的不足，引出我们的差异化

**现有方案对比：**

| 维度 | 传统BI工具 | 单一LLM方案 | 本项目（幻流） |
|------|-----------|------------|-------------|
| 数据理解 | 结构化查询为主 | 通用文本生成 | RAG精准检索+结构化数据融合 |
| 知识更新 | 依赖人工维护 | 训练数据截止 | 实时联网搜索+数据源自动更新 |
| 专业深度 | 预设模板 | 幻觉风险高 | 多Agent分工+来源引用可溯 |
| 用户适配 | 功能固定 | 一刀切 | 5种角色×9模块，千人千面 |
| 可解释性 | 仅图表展示 | 黑盒推理 | 推理链路可视化+引用来源标注 |

**核心技术挑战：**
1. 金融文档结构复杂（含表格、公式、脚注），PDF解析精度要求高
2. 领域知识密度大，通用LLM幻觉风险显著
3. 需要同时处理结构化（行情数据）+ 非结构化（PDF/新闻）数据
4. 不同层级用户需求差异大，需灵活的功能编排能力

## 第9页 · 核心技术①：RAG 检索增强生成

> 本项目最核心的技术亮点

### RAG Pipeline 全流程

```
数据源层                预处理层              向量数据库
┌─────────┐      ┌──────────────┐     ┌──────────────┐
│ PDF财报  │──→  │ PDF解析       │──→  │              │
│ SQL数据  │──→  │ (表格/文本/   │──→  │  Embedding   │
│ Wiki知识 │──→  │  OCR提取)     │──→  │  向量化存储   │
│ 研报文档  │──→  │ 滑动窗口分块  │──→  │  (3072维)    │
└─────────┘      │ overlap=64   │     └──────┬───────┘
                  └──────────────┘            │
                                              ↓
用户查询                                  ANN近似检索
┌─────────┐     ┌──────────────┐     ┌──────────────┐
│ 自然语言 │──→  │ 意图识别      │──→  │ 余弦相似度    │
│ 问题     │──→  │ Query改写     │──→  │ Top-K召回    │
└─────────┘     │ 关键词提取    │     │ 语义剪枝     │
                 └──────────────┘     └──────┬───────┘
                                              ↓
                  生成层                  上下文注入
                 ┌──────────────┐     ┌──────────────┐
                 │ LLM生成答案   │←── │ Prompt模板    │
                 │ +来源引用标注  │     │ +检索片段注入  │
                 └──────────────┘     └──────────────┘
```

### 关键技术细节

| 环节 | 技术选型 | 说明 |
|------|---------|------|
| PDF解析 | Dify内置解析器 + OCR | 支持表格、公式、脚注提取 |
| 分块策略 | 滑动窗口 512 tokens, overlap 64 | 保证上下文连贯性 |
| Embedding | text-embedding-3-large | 3072维高精度向量 |
| 向量存储 | Dify 内置 Vector DB (Weaviate/Qdrant) | ANN索引加速检索 |
| 检索策略 | 混合检索(向量+关键词) + 语义重排 | 兼顾精确匹配与语义理解 |
| 生成约束 | 引用来源标注 + 置信度阈值 | 降低幻觉风险 |

### 性能指标

| 指标 | 数值 |
|------|------|
| 检索召回率 (Recall@10) | **98.3%** |
| Top-5 精确率 (Precision@5) | **94.7%** |
| 首Token延迟 (P95) | **< 1.2s** |
| PDF解析成功率 | **99.1%** |

> 测试集：200条金融领域QA对，覆盖财务指标查询、趋势分析、对比分析等场景

## 第10页 · 核心技术②：多Agent协同架构

> 对标赛题"智能体赋能"核心要求

### Agent 角色分工

```
                    ┌──────────────────────┐
                    │   调度 Agent          │
                    │  (ChatPreprocessor)   │
                    │  意图识别 + 路由分发    │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              ↓                ↓                ↓
    ┌─────────────────┐ ┌──────────────┐ ┌──────────────┐
    │  RAG Agent       │ │ 分析 Agent    │ │ 生成 Agent   │
    │  (Dify引擎)      │ │ (Kimi k2.5)  │ │ (DeepSeek)   │
    │                  │ │              │ │              │
    │ · 财报问答       │ │ · 联网搜索    │ │ · 标题生成    │
    │ · 文档检索       │ │ · 行情数据    │ │ · 报告撰写    │
    │ · 来源引用       │ │ · 深度分析    │ │ · 数据摘要    │
    │ · 多文档对比     │ │ · 追问对话    │ │ · 模板填充    │
    └─────────────────┘ └──────────────┘ └──────────────┘
              │                │                │
              └────────────────┼────────────────┘
                               ↓
                    ┌──────────────────────┐
                    │  结果聚合 + 流式输出    │
                    │  SSE StreamHub        │
                    └──────────────────────┘
```

### 关键设计

- **意图路由：** ChatPreprocessor 根据用户输入自动识别意图（财报问答/行情分析/通用对话），路由至对应Agent
- **互斥锁：** Redis分布式锁（TTL 5min防死锁），同一股票同时只有一个分析任务
- **结果缓存：** 分析结果 Redis 缓存10min，避免重复API调用
- **流式输出：** SSE分阶段推送（数据采集→联网搜索→分析生成），用户实时看到进度
- **异步执行：** 后台线程执行耗时分析，不阻塞主请求循环

### 与赛题"智能体"要求对标

| 赛题要求 | 实现方式 |
|---------|---------|
| 以智能体为核心载体 | 多Agent架构，各Agent独立决策 |
| 自然交互 | 多轮对话 + 流式响应 + 意图澄清 |
| 自主分析 | Kimi Agent自动决定是否联网搜索 |
| 多源数据整合 | Tushare行情 + 财报PDF + 新闻舆情 融合输入 |

## 第12页 · 核心技术④：智能对话与可解释推理

> 对标赛题"需求澄清、多轮沟通"和"来源追溯、归因分析"

### 智能对话流程

```
用户输入 → 意图识别(ChatPreprocessor)
           │
           ├─→ 财报问答意图 → 检查工作区文档 → Dify RAG对话
           │                                    └→ 流式输出 + 来源引用
           ├─→ 行情分析意图 → Agent分析(Kimi)
           │                    └→ 数据采集→联网搜索→生成报告
           ├─→ 通用咨询意图 → DeepSeek直接回复
           │
           └─→ 澄清请求 → 多轮追问引导 → 明确意图后路由
```

### 可解释性设计

1. **来源引用标注：** RAG回答自动标注引用的PDF页码和原文片段
2. **推理链路可视化：** 前端展示Agent每一步决策过程（调用了什么工具、搜索了什么、推理了什么）
3. **置信度展示：** 检索结果附带相似度分数，低置信度时主动告知用户
4. **数据溯源：** 每个数据点标注来源（Tushare/AKShare/PDF/新闻）

### 对话能力指标

| 能力 | 说明 |
|------|------|

## 第13页 · 技术创新①：混合检索 + 语义重排

> RAG领域的创新设计

### 传统方案 vs 本项目

| 环节 | 传统方案 | 本项目 |
|------|---------|-------|
| 分块 | 固定长度切分 | 滑动窗口(512/64) + 语义段落感知 |
| 检索 | 纯向量检索 | **混合检索(向量+BM25关键词)** |
| 重排 | 无 | **语义重排(Cross-Encoder)** |
| 过滤 | Top-K截断 | **置信度阈值+语义剪枝** |

### 混合检索策略

```
用户Query
    │
    ├─→ 向量检索 (Embedding相似度)  → 候选集A (语义匹配)
    │
    ├─→ 关键词检索 (BM25)           → 候选集B (精确匹配)
    │
    └─→ 融合策略 (RRF倒数排序融合)   → 合并候选集
                                          │
                                    语义重排 (Cross-Encoder)
                                          │
                                    Top-K + 置信度过滤
                                          │
                                     最终检索结果
```

**为什么有效：**
- 向量检索擅长语义理解（"公司赚了多少钱" → 匹配"净利润"）
- 关键词检索擅长精确匹配（"ROE" → 精确命中）
- RRF融合取两者之长，显著提升召回率
- Cross-Encoder重排进一步提升精确率

> **开发计划：** Dify已内置混合检索能力，仅需在知识库配置中开启"混合检索"模式并调参。实现成本低，预计 0.5 天。

## 第14页 · 技术创新②：Agent自主决策与工具调用

> 展示Agent智能程度

### Kimi k2.5 builtin $web_search

本项目利用 Kimi 官方的 `builtin_function.$web_search`，模型在分析过程中**自主决定是否需要联网搜索**，而非硬编码搜索逻辑。

```
分析流程：
  1. 注入 Tushare 行情数据作为上下文
  2. 模型阅读数据后自主判断：
     - 数据充分 → 直接生成分析
     - 需要更多信息 → 自动调用 $web_search
       → 搜索最新新闻/公告/行业动态
       → 将搜索结果融入分析
  3. 多轮tool_call循环（最多6轮），直到模型认为信息充分
  4. 生成最终分析报告
```

### Agent决策透明化

```json
// 实际日志示例
[Agent] $web_search round 1: {"query": "比亚迪 2024年Q4 销量"}
[Agent] $web_search round 2: {"query": "新能源汽车 补贴政策 2025"}
[Agent] 分析完成，共搜索2轮，生成报告 3,200 tokens
```

前端推理可视化模块将上述决策链路以**节点图**形式展示给用户。

> **开发计划：** 已实现核心Agent调用链。推理可视化前端节点图已有基础组件（logic模块），需完善数据传递。预计 1-2 天。

## 第15页 · 技术创新③：多角色自适应架构

> 对标赛题"考虑不同用户需求"

### 设计理念：同一平台，千人千面

```
用户注册/登录
      │
      ↓
  角色选择（5选1）
      │
      ↓
  模块权限计算 ← ROLES 配置表
      │                │
      ↓                ↓
  前端模块动态加载    功能粒度控制
  (仅渲染可见模块)   (同一模块内子功能按角色过滤)
      │
      ↓
  localStorage 持久化
  (支持用户自定义调整)
```

### 三级权限粒度

| 级别 | 说明 | 实现方式 |
|------|------|---------|
| 模块级 | 角色决定可见模块集合 | `ROLES[role].defaultModules` |
| 功能级 | 同一模块内不同子功能 | `feature.roles.includes(role)` |
| 数据级 | 不同角色看到不同深度的分析 | API层按角色返回不同粒度数据 |

### 用户角色详情

| 角色 | 典型用户 | 核心价值 |
|------|---------|---------|
| 大众用户 | 股民小白 | 多轮问询降低门槛，享受分析师级服务 |
| 高净值用户 | 百万级投资者 | 深度资产配置，个性化投资建议 |
| 专业投资者 | 基金经理/分析师 | 产业链分析，一键研报，跳过小白引导 |
| 中小企业 | 创业公司CFO | 竞品分析，行业对标，轻量RAG检索 |
| 大型企业 | 券商/银行 | 全模块开放，多源异构数据接入，私有化部署 |

> **开发计划：** 前端角色选择UI和模块过滤逻辑已完成。后端API层角色过滤需新增中间件，预计 1 天。
