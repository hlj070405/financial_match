核心挑战：一次对话中协调多个异构工具并全程流式输出

用户的一个问题可能同时需要向量库检索（查已有财报）、财报抓取入库（缺失时自动下载）、量化回测（执行策略）、虚拟交易（记录持仓）四类异构工具协同工作。工具调用发生在流式对话过程中，模型需要自主决定调用哪些工具、以什么顺序调用，且全程保持 SSE 流式推送 — 用户在等待工具执行时需要看到实时进度反馈（"正在检索知识库…""正在回测 贵州茅台「双均线策略」…"），而不是长时间的空白等待。

针对上述挑战，我们在 rag_chat_stream() 中实现了一个多轮 Agent Loop：先通过 retrieve() 获取初始上下文，然后进入 K2.5 function calling 循环 — 模型自主决定调用哪些工具，工具执行结果注入 messages 后模型继续推理，直到产生最终回答。全程通过 SSE 事件流实时推送进度、来源、文本等多种事件类型，确保用户体验流畅。核心架构如下：

Agent Loop 中模型可以调用的工具共四类，覆盖从知识检索到量化交易的完整业务链条。每个工具在执行过程中通过 progress_queue 实时推送进度事件，让用户始终知道系统在做什么：

search_vector_store向量库精确检索

功能：在用户知识库中检索与 query 最相关的文档块
调用时机：模型判断需要更多上下文、或换一个检索角度
参数：query + 可选 document_ids（限定范围）
返回：匹配的 chunks + sources 来源信息
频率控制：system prompt 限制"每家公司最多 1-2 次"

fetch_financial_report巨潮资讯网抓取 + 自动入库

功能：从巨潮资讯网下载指定财报 PDF → 解析 → 分块 → 入库 ChromaDB
调用时机：知识库中缺少用户问到的财报
参数：company_name + stock_code + year
返回：already_exists / success / failed + document_id
联动：入库后自动将 document_id 加入 preferred_document_ids

run_backtest量化回测引擎

功能：将自然语言策略翻译为条件 JSON → FlexibleStrategy 执行回测
调用时机：用户描述交易策略、问技术面分析
参数：ts_code + buy/sell_conditions + logic
返回：收益率 / 夏普比 / 最大回撤 / 交易明细 / 资金曲线
执行方式：run_in_executor 线程池（CPU 密集型）

portfolio_trade / portfolio_query虚拟持仓操作

trade：记录虚拟买入/卖出（自动计算 A 股佣金+印花税）
query：查询当前持仓 + 实时估值（联动 BaoStock 最新行情）
调用时机：用户说"买入 XX 股"或问"我的持仓"
返回：持仓详情 / 交易确认 / 盈亏统计
持久化：SQLAlchemy Portfolio + PortfolioTransaction 模型

在实际开发中，Agent 多工具协同的主要工程难点集中在流式与工具调用的混合时序控制、断线恢复、以及防止模型陷入工具调用死循环。以下是我们遇到的三个关键挑战及对应的解决方案：

⚙ 设计难点与解决思路

❶ 流式与 tool_calls 的混合时序
tool_calls 轮次中模型可能先输出一段"中间思考"文本再产生工具调用。解法：用 is_tool_call 标记位，一旦检测到 tool_calls 立即停止输出 content（避免泄漏中间推理），该轮 content 仅用于 messages 上下文传递
❷ 断线续传（resume）
移动端网络不稳定可能中断 SSE 连接。解法：StreamHub 为每个会话缓存最近 5000 条事件（带递增 event_id），客户端重连时携带 last_event_id，stream_from() 从断点处恢复推送
❸ 工具调用死循环防护
LLM 可能反复调用 search_vector_store 却找不到结果。解法：双重防护 — ① max_rounds 硬性限制循环次数，② system prompt 中明确规定"每家公司最多 1-2 次 search"，轮次耗尽则强制生成最终回答
