多用户数据隔离与实时行情缓存

Multi-User Isolation & Real-Time Market Cache · ChromaDB + Redis + Agent Mutex
⚠

核心挑战：多用户环境下的数据安全、缓存一致性与并发控制

系统需要服务多个用户，每个用户的财报文档、向量数据、持仓记录必须严格隔离；行情数据需要在实时性与 API 调用成本之间取得平衡；多个用户可能同时请求分析同一只股票，需要避免重复计算和资源浪费。此外，Tushare 返回的 pandas DataFrame 中包含 np.int64、np.float64、NaN 等类型，无法直接 JSON 序列化写入 Redis。

针对上述挑战，我们从三个维度构建了完整的多用户基础设施：数据隔离层确保用户数据不交叉泄露，分层缓存层在实时性与成本之间取得平衡，分布式锁层防止并发分析造成资源浪费。三个层面各自独立又协同工作，共同支撑多用户并发场景下的数据安全与系统性能。

上述三层基础设施在实际运行中相互配合，但仍有几个关键工程问题需要注意。以下是我们在多用户并发场景下遇到的三个典型挑战及对应的解决方案：

⚙ 设计难点与解决思路

❶ numpy 类型的 JSON 序列化
Tushare 返回 DataFrame 含 np.int64、NaN、pd.NaT 等类型，直接 json.dumps 会抛 TypeError。解法：_sanitize_records() 逐字段类型判断，6 种 numpy 类型统一转为 Python 原生类型，NaN/Inf/NaT 转为 None
❷ 缓存一致性 vs API 成本
传统 TTL 过期会导致大量无意义的 API 重复请求。解法：采用 Stale-While-Revalidate 模式 — 数据永久存储，仅在 _is_stale() 判定过时时才触发刷新，且过时期间旧数据继续服务，API 返回空时不覆盖旧数据
❸ 分布式锁的死锁防护
如果持锁线程崩溃，锁永远不释放。解法：SET NX EX 300 设置 5 分钟自动过期；分析完成后 finally 块中主动 DELETE 释放锁；错误结果缓存 1 分钟（vs 正常 10 分钟），让用户快速重试
