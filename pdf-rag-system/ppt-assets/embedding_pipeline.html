⚠

核心挑战：大批量 chunk 的 Embedding 效率与可靠性

单次 PDF 入库可产生 940+ 个 chunk，需要批量调用硅基流动 Embedding API。但 API 存在速率限制（并发请求数受限）和 token 限制（bge-large-zh 限制 512 tokens），而金融表格的 token 密度极高（大量数字和标点），600 个中文字符可能已超过 512 token 限制。串行调用 940 个 chunk 耗时不可接受，且任何一个批次失败都不应影响整体入库流程。

针对上述挑战，我们在 _get_embeddings() 中设计了三级容错的异步并发架构：通过信号量控制 5 路并发加速批量请求，两级截断策略应对 token 超限，零向量填充保障批次间索引对齐。同时通过 SiliconFlowEmbedding 适配类解决 ChromaDB 同步接口与 async 服务之间的桥接问题。完整数据流如下：

上述数据流中的每个关键环节分别对应三个核心设计：并发控制确保不触发 API 限流，二级截断容错处理 token 超限，零向量填充保障批次独立性。三者协同构成完整的容错体系：

1

并发控制Semaphore(5) + asyncio.gather

信号量：asyncio.Semaphore(5) 限制最多 5 个并发请求
并发执行：asyncio.gather(*tasks) 所有批次并行启动
批次大小：EMBEDDING_BATCH_SIZE=32（每请求 32 条文本）
超时保护：httpx.AsyncClient(timeout=60s)
吞吐量：940 chunks → ~30 批 → 5 路并发 → 6 轮即可完成
加速比：相比串行约 5x 提速
2

二级截断容错600字符 → 400字符 降级

一级：入口处所有文本截断到 600 字符（保守预估 ≈ 512 token）
触发二级：API 返回 413 或 20042（token 超限）
二级：该批次逐条截断到 400 字符 后重试
设计原因：金融表格数字密度高，600 字符可能实际超 512 token
自动降级：仅超限批次触发，其余批次不受影响
二次失败：标记为 None → 零向量填充
3

零向量填充批次独立 · 索引对齐

场景：某批次两次重试均失败
策略：用 [0.0] × 1024 零向量填充该批次所有位置
目的：保持向量列表与 chunks 列表索引一一对齐
影响：零向量在 cosine 相似度中距离任何查询都远，不会被检索命中
日志：打印警告 "N 条使用零向量填充"
优势：单批次失败不影响其余 900+ chunks 正常入库

除了批量 Embedding 本身的容错，还有一个关键工程问题需要解决：ChromaDB 的 EmbeddingFunction.__call__ 是同步接口，但我们的 RAG 服务运行在 async 环境中。SiliconFlowEmbedding 适配类通过运行时环境检测实现了无缝桥接：

def __call__(self, input): loop = asyncio.get_running_loop() if loop.is_running(): # 异步环境 → 线程池桥接 with ThreadPoolExecutor() as pool: future = pool.submit( self._sync_embed, input ) return future.result(timeout=120)

上述容错架构在实际运行中表现稳定，但仍有几个工程难点需要注意。以下是我们在开发和调优过程中遇到的三个关键问题及对应的解决方案：

⚙ 设计难点与解决思路

❶ token 限制的不可预测性
bge-large-zh 限制 512 token，但金融表格中数字和标点的 token 密度远高于普通文本（"130,456,789.00" 可能消耗 6+ token）。解法：保守截断到 600 字符（普通文本安全），超限时自动降级到 400 字符，两级兜底覆盖极端情况
❷ 同步/异步环境的无缝切换
ChromaDB 的 EmbeddingFunction.__call__ 是同步接口，而 _get_embeddings 是 async。解法：运行时检测 asyncio.get_running_loop()，异步环境用线程池+新事件循环桥接，同步环境直接 asyncio.run()
❸ 批次失败的隔离性
如果直接跳过失败批次，后续向量与 chunk 列表索引错位，ChromaDB 会将错误的向量匹配到错误的文本。解法：零向量填充保持索引一一对齐，零向量在 cosine 空间中与任何查询的相似度都极低，不会被误检索
