三阶段递进式检索漏斗架构

Three-Stage Progressive Retrieval Funnel · query_router.py + rag/service.py
⚠

核心挑战：金融财报领域的实体混淆问题

用户问"工商银行2024年营收"，纯向量检索可能返回招商银行或建设银行的数据 —— 银行类财报语义高度相似（余弦相似度 >0.85），仅靠 Embedding 无法区分实体边界。传统方案要么牺牲召回率（硬过滤），要么引入大量噪声（纯向量）。

针对上述挑战，我们在检索链路中设计了三阶段递进式漏斗架构：先通过意图解析锁定文档范围，再用 BM25 混合重排补充关键词精确匹配能力，最后由 Cross-Encoder 精排模型做最终相关性判定。三阶段逐层收窄候选池，在不牺牲召回率的前提下大幅提升精确率。具体架构如下：
S1
粗筛层

意图解析 + 元数据过滤 Intent Parsing & Metadata Filtering parse_intent() + build_where_filter()

输入： 用户自然语言 query（如"宁德时代2024三季报营收"）
公司识别： 别名词典（工行→工商银行、茅台→贵州茅台等）+ 6位股票代码正则匹配
时间解析： 年份20\d{2} + 季度映射（一季报→Q1、半年报→H1、年报→Q4）
过滤构建： 匹配 ChromaDB source 字段 → 生成 {"source": {"$in": [...]}} where 条件
优先级： 显式 document_id > 意图匹配 source > 全库检索（无意图时降级）
技术亮点： 零外部依赖，纯正则+字典，延迟 <1ms，无 LLM 调用开销

S1 完成后，系统已将检索范围从全库缩小到目标文档。接下来通过向量检索获取语义相关的候选块，并进入第二阶段进行混合重排：

↓
向量检索 + 阈值过滤（score ≥ 0.55） → 候选池 ~20-40 条
↓
S2
混排层

BM25 混合重排 Hybrid Reranking with RRF Fusion hybrid_rerank()

向量路： BGE-large-zh-v1.5 余弦相似度，捕捉语义近似（"营业收入"≈"总收入"）
BM25路： 自研轻量 BM25（k1=1.5, b=0.75），逐字分词+英文token，捕捉精确关键词
融合算法： RRF（Reciprocal Rank Fusion）: score = 0.6/(k+rank_vec) + 0.4/(k+rank_bm25), k=60
归一化： BM25 分数归一化至 [0,1]，向量分本身为余弦相似度 [0,1]
权重分配： 向量权重 0.6 : BM25 权重 0.4（金融语料实验调优值）
技术亮点： 零外部依赖的 BM25 实现，无需 Elasticsearch，轻量嵌入后端

S3
精排层

BGE-Reranker 精排 + 断路器 Cross-Encoder Reranking & Circuit Breaker bge_rerank()

模型： BAAI/bge-reranker-v2-m3（Cross-Encoder 架构，硅基流动 API 托管）
原理： 将 query + document 拼接后整体输入，注意力机制同时感知两者的交互关系
断路器： 最高 relevance_score < 0.6 → 触发断路，标记 is_relevant=False
降级策略： API 超时/Key缺失时自动降级为 S2 排序结果，保障服务可用性
输出控制： 按 rerank_score 降序 → 截取 Top-K（默认5条）返回给 LLM 生成层
技术亮点： 精准判断 query-document 相关性，比 Bi-Encoder（向量）精度提升 15-30%

在实际开发过程中，三阶段架构的参数调优与降级策略是主要的工程难点，以下是我们遇到的关键问题及对应解决思路：

⚙ 设计难点与解决思路

❶ 权重如何调优？
BM25权重 0.4 与向量权重 0.6 基于金融语料 A/B 实验确定；过高的 BM25 权重会导致同义词（"营业收入"vs"总收入"）漏召回
❷ 断路器阈值设定？
阈值 0.6：过高会误杀相关文档（假阴性），过低则放行噪声。金融报表语义密度高，需要比通用场景更严格的阈值
❸ 三阶段串行延迟？
S1+S2 均为本地计算（<6ms），延迟主要来自 S3 API 调用（~200ms）；通过 S1 缩小候选池，有效控制 S3 的 payload 大小
