金融PDF智能分块架构

Financial PDF Intelligent Chunking · chunker.py + rag/service.py
⚠

核心挑战：财报 PDF 的结构复杂性导致传统分块质量极差

财报 PDF 存在表格跨页（资产负债表常跨3-5页）、章节嵌套（"第五节 重要事项 → 一、主营业务 → (一)分行业"三级标题）、扫描件混排（文字页与图片页交替）等问题。传统固定字符切分（如512字符滑窗）会导致表格被拦腰切断、数字与描述分离（"营业收入"在上一个chunk，"130,456,789.00"在下一个chunk）、上下文语义断裂。

针对上述问题，我们在 chunker.py 中设计了三策略联合的智能分块方案：章节标题感知切分、表格整体保留、跨页表格修复，三者协同工作以保障每个 chunk 的语义完整性。同时，在 PDF 解析层实现了双轨自适应策略，根据文件大小自动选择精度优先或效率优先的解析模式，如下所示：
策略切换条件：页数 ≤ 80 → Layout模式 ｜ 页数 > 80 → 快速模式（convert_pdf_to_markdown）

Layout 结构化模式 pymupdf4llm · 小文件路径 ≤ 80页

解析引擎：pymupdf4llm Layout ON
输出格式：结构化 Markdown（#/##/### 层级标题 + MD表格）
表格处理：原生识别表格边界，输出 | col1 | col2 | 格式
优势：章节层级完整保留，表格结构精确
代价：417页财报耗时 ~100s（CPU密集型）
分块器：MarkdownHeaderTextSplitter 按 #/##/### 拆分
OR

快速文本+表格模式 PyMuPDF text + pdfplumber tables · 大文件路径 > 80页

文本引擎：PyMuPDF page.get_text("text") 逐页提取
表格引擎：pdfplumber 前 30 页表格提取（避免超时）
输出格式：## 第N页 分隔的纯文本 + 附录表格区
优势：417页财报仅 ~10s，速度提升 10倍
代价：丢失标题层级，表格仅覆盖前30页
分块器：_split_into_sections 章节识别 + 固定兜底

完成 PDF 解析后，系统将提取到的文本送入智能分块引擎。针对金融财报的特殊结构，我们设计了三种互补的切分策略，在 auto 模式下自动协同工作：

1

章节标题感知切分Section-Aware Splitting

核心函数：_score_section_title()
原理：对每行文本多维评分，≥3分判定为标题行
标题边界：按标题行切分为逻辑段落
超长兜底：段内超过 512 字符 → RecursiveCharacterTextSplitter 二次切分
分隔符优先级：\n\n > \n > 。 > ； > 空格
上下文注入：每 chunk 头部追加 当前章节：{title}
2

表格整体保留Table Preservation

检测方式：is_table 标记 或 MD 表格语法 |-|
保留规则：表格内容不拆分，整体作为单个 chunk
超长截断：超过 chunk_size × 4（2048字符）时截断并标注
表格判定：行内含 ≥3 个纯数字 token 或 |/\t 分隔符
数字密度：数字字符占比 >50% 的行视为数据行
元数据标注：is_table: True + section_title: "表格数据"
3

跨页表格修复Cross-Page Table Repair

核心函数：_repair_table_headers_across_pages()
问题场景：## 第N页 将表头和数据体分到不同 section
表头检测：页末短行（≤40字）含"项目/年/季度/期末"等9个标记词
数据验证：下页开头存在 ≥2 行数字密集行时才执行修复
修复动作：复制表头行到下一页开头，保持表格可读性
防误判：表头候选 ≥2 行 + 数字占比 ≤50% 双重校验

其中，策略一的章节标题识别是整个分块质量的关键 — 如何在金融文档中准确区分"标题行"和"数据行"？我们设计了一套多维评分机制，通过正向加分与负向惩罚的组合来可靠判定：

标题评分机制详解 _score_section_title() · 阈值 ≥ 3分

评分规则 分值 说明 示例
编号前缀（"1." / "第一节" / "(一)"） +3 检测数字编号、中文序号、括号序号三种格式 "1. 营业收入" → +3
短行（长度 ≤ 20字符） +2 标题通常较短；21-32字 +1分 "主营业务分析" (7字) → +2
标点符号少（≤1个标点） +1 标题极少包含逗号、句号等正文标点 "财务报表附注" → +1
下一行更长（长度差 >10字） +1 标题后通常跟随较长的正文段落 下一行 50字 vs 当前 12字 → +1
以句号/逗号/分号结尾 -2 强负信号：标题几乎不以标点结尾 "报告期内主要经营如下。" → -2
数字占比超过 35% -2 过滤数据行（如 "130,456,789.00 元"） "营业收入 130,456,789.00" → -2
看起来像表格行（≥3个数字 token） -3 最强负信号：表格数据行绝不是标题 "工商银行 | 8,567 | 12.3%" → -3
判定阈值：总分 ≥ 3 → 标题行 ｜ 前置过滤：长度 >40字 / 纯数字 / 无汉字字母 → 直接排除（-10分）

在实际工程中，上述方案的难点主要集中在启发式规则的边界情况处理上。以下是我们遇到的三个关键挑战及对应的解决思路：

⚙ 设计难点与解决思路

❶ 标题 vs 数据行的模糊边界
金融文档中"1. 营业收入 130,456,789.00"既像标题又像数据行。解法：数字占比惩罚（>35%扣2分）+ 表格行检测（≥3个纯数字token扣3分），多维评分互相校正
❷ 跨页修复的误判控制
非表格的短行也可能被误判为表头。解法：双重验证 — 上页末尾需含 ≥2 行候选 + 9个标记词命中，下页开头需有 ≥2 行数字密集数据行，才执行修复
❸ 大小文件策略平衡
Layout模式精度高但417页需~100s，纯文本仅~10s但丢结构。解法：80页阈值自适应切换 + pdfplumber 补充前30页关键表格，兼顾精度与效率
