核心挑战：自然语言交易策略的结构化翻译与精确执行

用户用自然语言描述交易策略（如"布林带下轨抄底且放量确认"、"5日线上穿20日线就买"），系统需要准确理解策略意图，翻译为可执行的量化条件，并在回测引擎中精确运行。难点在于：用户表述模糊（"均线策略"没指定哪根线）、条件不完整（只说买入没说卖出）、交叉信号的数学定义需要严格实现（前一根K线和当前K线的相对位置判断）。

针对上述挑战，我们设计了 K2.5 Agent + FlexibleStrategy 协同架构：由 LLM 负责将自然语言翻译为结构化条件 JSON，由 FlexibleStrategy 解析条件并驱动 Backtrader 引擎执行回测。LLM 翻译层通过详尽的 system prompt 和翻译示例保障准确性，策略执行层通过统一的表达式解析器和指标缓存机制保障可靠性。整体数据流与核心模块如下：

上述数据流的每个环节分别由三个核心模块承担：LLM 翻译层负责语义理解与结构化转换，策略解析层负责表达式解析与指标计算，回测执行层负责真实费用模拟与统计分析。三层各司其职、逐层解耦：

1

LLM 翻译层K2.5 Agent · System Prompt

触发方式：K2.5 Agent 识别回测意图 → 自动调用 run_backtest Tool
翻译目标：自然语言 → buy_conditions + sell_conditions JSON
默认值兜底：均线默认 sma(5)/sma(20)，RSI 默认 14/30/70
条件补全：用户只说买入 → LLM 自动补全对称卖出条件
Prompt 工程：内嵌 4 组翻译示例（SMA/RSI/布林带/MACD）确保输出格式一致
多股对比：行业级问题自动选 3-5 只代表个股分别回测
2

策略解析层FlexibleStrategy · _parse_expr

核心类：FlexibleStrategy(bt.Strategy)
表达式解析：_parse_expr("sma(5)") → ("sma", [5]) 元组
指标缓存：_ind_cache 按需创建 Backtrader 指标对象，避免重复计算
值获取：_get_val(expr, offset) 统一处理指标/价格/常量
条件求值：_eval_cond() 支持 6 种运算符
逻辑组合：_check_conditions() 支持 all(AND) / any(OR)

交叉信号实现详解 cross_above / cross_below · _eval_cond()

cross_above（金叉）
# 前一根K线: 左 ≤ 右（在下方） prev_l = _get_val(left, -1) prev_r = _get_val(right, -1) # 当前K线: 左 > 右（穿越到上方） return prev_l <= prev_r and left > right
场景：SMA(5) 从下方穿越 SMA(20) → 买入信号
VS
cross_below（死叉）
# 前一根K线: 左 ≥ 右（在上方） prev_l = _get_val(left, -1) prev_r = _get_val(right, -1) # 当前K线: 左 < 右（穿越到下方） return prev_l >= prev_r and left < right
场景：SMA(5) 从上方穿越 SMA(20) → 卖出信号

在实际工程中，上述方案的核心难点集中在 LLM 翻译的鲁棒性和交叉信号的精确实现上。以下是我们遇到的三个关键挑战及对应的解决思路：

⚙ 设计难点与解决思路

❶ LLM 翻译的模糊性兜底
用户说"均线策略"没指定哪根线，说"抄底"没给阈值。解法：system prompt 内嵌 4 组完整翻译示例 + 行业惯例默认值（均线 5/20、RSI 14/30/70），确保 K2.5 在信息不全时仍能生成合理条件
❷ 买卖条件的不对称补全
用户只说买入没说卖出，策略无法闭环。解法：system prompt 明确要求 "条件不完整时合理补全卖出条件"，如买入用 cross_above 则卖出用对称的 cross_below
❸ cross_above/below 的边界精度
交叉信号需对比前一根K线与当前K线的相对位置。边界条件 prev_l <= prev_r and left > right 必须用非严格不等号（<= 而非 <），否则两线重合时会漏判
