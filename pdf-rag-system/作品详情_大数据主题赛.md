## 三、作品描述文本

### 1. 作品简介

幻思·智能金融分析决策支持系统是一个面向个人投资者和金融从业者的AI驱动一站式金融分析平台。系统以RAG（检索增强生成）为核心引擎，融合Agent智能编排、量化回测、实时行情、虚拟持仓五大核心模块，实现从"数据采集→文档理解→智能问答→策略验证→交易模拟"的全链路闭环。

**核心能力：**

- **RAG智能问答**：上传企业财报PDF，系统自动解析、分块、向量化入库，通过三阶段检索漏斗（向量召回→BM25混合重排→Cross-Encoder精排）实现高精度文档问答，支持多文档对比分析
- **Agent工具编排**：基于Kimi k2.5的多轮function calling架构，Agent可自主决策调用6个专业工具（向量检索、财报抓取、量化回测、虚拟交易、持仓查询、联网搜索），一次对话完成复杂金融分析任务
- **智能量化回测**：用户用自然语言描述交易策略（如"5日均线上穿20日均线买入"），Agent自动翻译为条件JSON，调用Backtrader引擎执行历史回测，返回收益率、夏普比率、最大回撤等专业指标
- **实时行情终端**：集成Tushare Pro数据源，提供K线图、技术指标叠加（MACD/RSI/布林带）、资金流向分析，数据通过Redis持久化缓存
- **虚拟持仓交易**：支持模拟买卖操作，自动计算A股真实费率（佣金万2.5、印花税千0.5），采用加权平均成本法跟踪持仓盈亏

**技术亮点：**

- 三阶段RAG检索漏斗 + 断路器机制（低相关度自动切换联网补充）
- Markdown结构化分块 + 章节标题感知 + 表格整体保留
- 条件式策略引擎FlexibleStrategy，支持11种技术指标和6种运算符
- SSE流式输出 + Redis互斥锁 + 异步后台执行
- Docker Compose一键部署（MySQL + Redis + ChromaDB + 后端 + 前端）

### 5. 设计思路

#### 5.1 整体架构

系统采用前后端分离的B/S架构，以"AI Agent + RAG"为核心设计理念：

```
用户界面（Vue3 SPA）
    ↓ HTTP/SSE
FastAPI 后端 API 层
    ↓
┌─────────────────────────────────────────────────┐
│              核心服务层                           │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │ RAG引擎  │  │Agent编排 │  │回测引擎  │       │
│  │(检索增强) │  │(工具调用) │  │(Backtrader)│      │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘       │
│       │             │             │              │
│  ┌────┴─────┐  ┌────┴─────┐  ┌────┴─────┐       │
│  │ChromaDB  │  │Kimi LLM  │  │BaoStock  │       │
│  │(向量库)  │  │(联网搜索) │  │(历史数据) │       │
│  └──────────┘  └──────────┘  └──────────┘       │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │MySQL     │  │Redis     │  │Tushare   │       │
│  │(业务数据) │  │(缓存/锁) │  │(实时行情) │       │
│  └──────────┘  └──────────┘  └──────────┘       │
└─────────────────────────────────────────────────┘
```

#### 5.2 RAG检索流程设计

传统RAG系统仅依赖向量相似度检索，存在"实体混淆"问题（如查询比亚迪却返回宁德时代的内容）。本系统设计了三阶段检索漏斗解决该问题：

1. **意图解析 + 向量召回**：从用户查询中提取公司名/年份/季度，构建ChromaDB元数据where过滤条件，缩小检索范围后执行向量相似度搜索
2. **BM25混合重排**：对向量检索结果叠加自研BM25词频评分，通过RRF（Reciprocal Rank Fusion）融合两路排名，提升硬实体命中率
3. **Cross-Encoder精排 + 断路器**：调用BGE-Reranker-v2-m3对query-document做整体注意力评分；若最高分低于阈值，触发断路器——系统自动切换联网搜索补充信息

#### 5.3 Agent编排设计

Agent采用"LLM驱动的循环决策"架构：
- LLM根据用户问题和工具描述，自主决定调用哪些工具、传入什么参数
- 工具执行结果作为新消息反馈给LLM，LLM继续决策直到生成最终回答
- 支持一次对话中多次工具调用（如先抓取财报→入库→检索→回测→生成报告）
- 全程SSE流式输出，用户可实时看到Agent的思考和工具执行过程

#### 5.4 量化回测设计

核心创新是"自然语言→量化策略"的自动翻译：
- 用户说"当RSI低于30时买入，高于70时卖出"
- Agent将其翻译为条件JSON：`buy_conditions: [{"left":"rsi(14)","op":"<","right":"30"}]`
- FlexibleStrategy类动态创建Backtrader技术指标对象，逐K线求值条件并执行交易
- 回测引擎模拟A股真实费率环境（佣金万2.5、卖出印花税千0.5、100股整手限制）

### 6. 设计重点难点

#### 重点一：RAG检索精度——解决"实体混淆"问题

**问题**：用户问"比亚迪2024年营收"，向量检索可能返回宁德时代的财报片段（因为两者在"新能源"语义空间距离近）。

**解决方案**：
- 查询路由层先提取意图（公司名→规范化、年份、季度），构建元数据过滤条件
- 向量检索限定在目标公司文档范围内，避免跨公司污染
- BM25补充词频匹配（"比亚迪"这种硬实体在BM25中权重极高）
- Reranker做最终语义精排，不相关内容被断路器拦截

#### 重点二：Agent多工具协作——从对话到闭环

**问题**：用户一句话可能包含多个子任务（"帮我分析宁德时代最新财报，顺便用MACD策略回测一下"），需要Agent自主拆解并依次完成。

**解决方案**：
- 6个工具通过JSON Schema严格定义参数格式和描述
- 系统提示词详细规范工具调用判断逻辑（何时调用fetch_financial_report vs search_vector_store）
- Agent循环最多15轮，每轮处理LLM返回的tool_calls，将执行结果注入消息历史
- 支持工具调用碎片拼装（LLM流式返回时参数可能跨多个chunk）

#### 重点三：自然语言→量化策略翻译

**问题**：用户描述策略的方式千变万化（"均线金叉"、"布林带下轨抄底"、"放量突破"），需要可靠地翻译为程序可执行的条件。

**解决方案**：
- 定义标准化的条件格式：`{left: 指标表达式, op: 运算符, right: 指标表达式或数值}`
- 支持11种指标表达式：price, sma(N), ema(N), rsi(N), macd(), macd_signal(), boll_upper(N), boll_lower(N), boll_mid(N), volume, volume_ma(N)
- 支持6种运算符：>, <, >=, <=, cross_above, cross_below
- FlexibleStrategy在`__init__`时动态创建所有需要的Backtrader指标对象，`next()`中逐条件求值
- 系统提示词中包含详细的翻译原则和默认参数规范

#### 难点一：文档分块质量——财报PDF的结构化挑战

**问题**：财报PDF动辄数百页，包含大量表格、页眉页脚、跨页内容，简单的固定长度切分会破坏语义完整性。

**解决方案**：
- 优先使用pymupdf4llm将PDF转为结构化Markdown，保留标题层级和表格格式
- MarkdownHeaderTextSplitter按标题层级切分，确保同一章节内容不被拆散
- 章节标题评分函数（_score_section_title）通过编号前缀、长度、标点、数字占比等多维度识别真实标题
- 表格整体保留策略：检测到表格行特征后不切分，超长表格截断并标注
- 每个chunk注入上下文前缀（当前章节路径），确保脱离上下文后仍可理解

#### 难点二：流式响应架构——SSE + 工具调用的并发控制

**问题**：Agent对话需要同时支持流式文本输出和中间穿插工具调用，且工具执行可能耗时较长（如回测需要下载数据）。

**解决方案**：
- 使用asyncio.Queue作为进度通道，工具执行函数通过queue推送阶段性进展
- SSE事件分类：phase（阶段提示）、delta（文本增量）、tool_start/tool_result（工具过程）、done（完成）
- Redis分布式锁防止同一分析任务并发执行，超时自动释放防死锁
- 指数退避重试处理LLM引擎过载（engine_overloaded）
